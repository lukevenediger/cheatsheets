cloudflare cheatsheet

Tunnels from home/desktop to the public internet — and Zero Trust Access + WAF to keep them locked down.

Pattern: cloudflared on the desktop creates an outbound mTLS connection to Cloudflare. Public DNS for app.yours.com CNAMEs to <uuid>.cfargotunnel.com. Add a Zero Trust Access policy on that hostname so requests must authenticate (Google / GitHub / one-time PIN) before the tunnel forwards them.

Install & auth

Tunnel lifecycle

cloudflared tunnel create home-desk cloudflared tunnel list cloudflared tunnel info home-desk cloudflared tunnel token home-desk # token-only mode cloudflared tunnel delete home-desk cloudflared tunnel cleanup home-desk # stale conns

Creates UUID + ~/.cloudflared/<uuid>.json credentials.

DNS routing

Tunnel config (~/.cloudflared/config.yml)

tunnel: 5f6e1b8e-...                  # UUID from `tunnel list`
credentials-file: /root/.cloudflared/5f6e1b8e-....json

# Optional: prefer the config file location for the service
# loglevel: info
# metrics: 0.0.0.0:9000

ingress:
  # multiple hostnames -> multiple local services
  - hostname: ssh.yours.com
    service: ssh://localhost:22
  - hostname: app.yours.com
    service: http://localhost:3000
    originRequest:
      noTLSVerify: true             # only when origin self-signed
      connectTimeout: 10s
  - hostname: rdp.yours.com         # public RDP via WARP client only
    service: rdp://localhost:3389
  # catch-all MUST be last
  - service: http_status:404

Each ingress entry maps one Cloudflare hostname to one origin service running on this machine. Service supports http://, https://, tcp://, ssh://, rdp://, unix:, hello-world.

Lock it down: Zero Trust Access

A bare tunnel exposes your desktop service to the open internet. Wrap each public hostname in a Zero Trust Access application so Cloudflare blocks unauthenticated traffic before it crosses the tunnel.

  1. Add identity provider: Zero Trust → Settings → Authentication → add Google, GitHub, One-time PIN, or SAML/OIDC.
  2. Create Access app: Zero Trust → Access → Applications → Add a self-hosted application. Set Application domain to app.yours.com (the same hostname your tunnel routes).
  3. Allow policy: action Allow, include Emails = me@yours.com (or Emails ending in @yours.com, or an Access Group).
  4. Block everything else: add a second policy with action Block, include Everyone, lower priority.
  5. Headless / API access: use a Service Auth token (Access → Service Auth) and pass CF-Access-Client-Id / CF-Access-Client-Secret headers.

For SSH/RDP, install the WARP client and use the Access for Infrastructure short-lived SSH cert flow — no static keys on the desktop.

Access policies (examples)

  • Allow my emails: Emails = me@yours.com, partner@example.com
  • Allow domain: Emails ending in @yours.com
  • Require MFA: add Require → Authentication method = mfa
  • Country fence: add Require → Country = AU, US
  • Device posture: require WARP client + disk encryption
  • Service token only: action Service Auth, include token
  • Session TTL: shorten to 1h on sensitive apps

WAF & rate limit

Layer in front of (or instead of) Access for public endpoints. Security → WAF in zone dashboard.

  • Managed Ruleset: enable Cloudflare Managed Ruleset + OWASP Core Ruleset — click toggle, set paranoia to 2.
  • Custom rule: (cf.threat_score gt 14) or (ip.geoip.country ne "AU") → Managed Challenge.
  • Bot fight: Security → Bots → toggle Bot Fight Mode (free) or Super Bot Fight Mode (paid).
  • Rate limit: Security → Rate limiting rules → e.g. http.request.uri.path eq "/login" → 5 req/min/IP → Block 10m.
  • API Shield: upload OpenAPI schema → auto-reject non-conforming requests.
  • Geo-block fast: ip.geoip.country in {"RU" "CN" "KP"} → Block.
